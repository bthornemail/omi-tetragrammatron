# OMI Doctrine: Notation as Cipher, Cipher as Notation

Lisp teaches:

```text
code as data
data as code
```

POSIX teaches:

```text
everything is a file
```

OMI teaches:

```text
notation as cipher
cipher as notation
```

---

## 1. The Difference Between Code and Cipher

Conventionally, code operates on meaning.

A code replaces:

```text
word
phrase
sentence
symbol
```

with another symbolic representation.

A cipher operates on structure.

A cipher rewrites:

```text
bits
bytes
characters
patterns
order
direction
surface
```

without requiring semantic understanding.

This distinction is fundamental:

```text
Code acts on meaning.
Cipher acts on form.
```

---

## 2. OMI Removes the Boundary

OMI removes the hard boundary between code and cipher.

In OMI:

```text
a notation is a cipher
a cipher is a notation
```

Changing notation changes interpretation.

Changing interpretation changes notation.

The binary source may remain unchanged.

Only the rewrite surface changes.

---

## 3. The Binary Source Does Not Have to Change

Traditional systems assume:

```text
data changes
```

OMI assumes:

```text
interpretation changes
```

The same binary source may be read as:

```text
text
address
instruction
receipt
graph
proof
nomogram
rewrite table
carrier
projection
```

depending on the notation applied.

Thus:

```text
notation = cipher
```

because notation rewrites interpretation.

---

## 4. Omicron as Cipher Selector

The Omicron is not merely a value.

The Omicron is a selector.

It selects the active notation surface.

```text
omi-
```

and

```text
-imo
```

are dual readings of the same structure.

The bits remain.

The interpretation rotates.

Canonical rule:

```text
OMI selects the reading.
IMO carries the inverse reading.
The source remains.
The notation changes.
```

---

## 5. The Rewrite Surface

OMI does not begin with objects.

OMI begins with rewrite surfaces.

A rewrite surface is any structure capable of changing interpretation:

```text
endianness
BCD
UTF
UTF-EBCDIC
escape sequences
BOM markers
control characters
frame headers
BiDi markers
private-use code points
carrier headers
```

Each acts like a cipher.

None has to modify the underlying source.

They modify how the source is read.

---

## 6. Control Characters as Cipher Operators

The ASCII control range:

```text
0x00..0x1F
```

is traditionally described as non-printing characters.

OMI treats these as notation operators.

They define transitions:

```text
start
stop
shift
escape
acknowledge
separate
return
advance
close
open
```

These are not merely symbols.

They are interpretation transitions.

They are cipher operations.

The readable boundary:

```text
0x20
```

opens the surface where prose, glyphs, and visible notation may begin.

Canonical rule:

```text
0x00..0x20 is the pre-language cipher envelope.
```

---

## 7. Unicode as Notation Space

Unicode is usually described as a character space.

OMI treats Unicode as notation space.

Each code point may become:

```text
a glyph
a boundary
a marker
a selector
a rewrite hint
an interpretation surface
```

Private-use regions are useful because they allow notation without conflicting with public meaning.

Canonical rule:

```text
Unicode is not only where characters live.
Unicode is where notation can be staged.
```

---

## 8. Cipher Without Secrecy

Cryptography usually uses ciphers for secrecy.

OMI uses cipher structure for interpretation.

The goal is not:

```text
hide meaning
```

The goal is:

```text
declare reading
```

The mechanism is related.

The purpose differs.

Canonical rule:

```text
OMI ciphers do not have to conceal.
They have to select and attest interpretation.
```

---

## 9. BiDi as Dual Reading

BiDi is not merely left-to-right versus right-to-left.

BiDi expresses a deeper principle:

```text
one structure may have multiple lawful readings
```

The same source may be read:

```text
forward
backward
mirrored
folded
projected
inverted
```

without changing the source itself.

This is the OMI principle in visible form:

```text
the source remains
the reading changes
```

---

## 10. Notation, Citation, Attestation, Attribution

OMI is not primarily a file format.

OMI is not primarily a barcode.

OMI is not primarily an image.

OMI is not primarily a programming language.

OMI is a protocol for:

```text
notation
citation
attestation
attribution
```

Notation selects the reading.

Citation locates the relation.

Attestation records the witnessed claim.

Attribution binds the claim to a source, role, or receipt path.

Canonical rule:

```text
Notation selects.
Citation locates.
Attestation witnesses.
Attribution binds.
Receipt accepts.
```

---

## 11. Why Extra Names Fall Away

During development, many names are useful:

```text
matrix
bitboard
image
barcode
carrier
adapter
profile
version
renderer
```

But these names are scaffolding.

Once OMI resolves itself, these names reduce to relations.

The protocol does not need a large vocabulary of carrier names.

It needs a small number of relation forms.

Development names may exist outside the protocol.

Inside OMI, they compile down to:

```text
(scope . relation)
(car . cdr)
(candidate . receipt)
(source . attribution)
(notation . cipher)
```

Canonical rule:

```text
Names help development.
Relations define the protocol.
```
